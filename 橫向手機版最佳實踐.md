# 橫向手機版（Landscape Mobile）最佳實踐

> 調研日期 2026-06-16。結論基於目前各大網站實際做法 + WCAG 2.1 規範 + 本專案實際 code 盤點。
> 核心精神（user 指示）：**不要從頭寫起，能用瀏覽器 / Tailwind 已經寫好的就用，改成我們自己的即可。**

---

## 0. 一句話結論

**各大網站「沒有」在做獨立的橫向手機版**——主流做法是「同一套響應式版面，讓它在橫向（寬而矮）的視窗也不會壞」。
真正要解的不是「做一個橫版」，而是**橫向手機的「高度」很矮**（約 375–430px），把為高螢幕設計的滿版 hero 壓壞。
所以本專案要做的不是「第二套版型」，而是**加一層「矮橫向」的防護 CSS**，用瀏覽器原生的 `svh/dvh` 單位 + Tailwind 內建的 `landscape:` variant，幾乎零自寫。

---

## 1. 各大網站現況：到底有沒有「橫向手機版」？

| 做法 | 誰在用 | 評價 |
|---|---|---|
| **純流式響應式**（橫向手機 = 較寬的視窗，沿用同一套斷點） | 絕大多數網站（Apple / Google / 媒體 / 電商 / 多數作品集站） | ✅ 業界標準。沒有「橫向專屬版型」這種東西 |
| **依高度微調**（`orientation: landscape` + `max-height` 收掉滿版高度） | 注重體驗的站、有滿版 hero 的站 | ✅ 最佳實踐，正是本專案需要補的 |
| **「請旋轉裝置」強制鎖方向**（pleaserotate.js 之類） | 少數遊戲 / 簡報 / 早期 RWD 站 | ❌ **反模式**，且違反 WCAG（見下） |
| **獨立的 landscape 專屬 layout** | 幾乎沒有 | ❌ 維護兩套版型，業界已淘汰 |

**重點：你不會找到「大網站做了一個橫向手機專版」的案例。** 他們做的是「讓既有響應式版面在矮橫向視窗也能用」。所以本專案的目標應對齊第二列——**補一層矮橫向防護**，而不是寫第二套版型。

---

## 2. 不可以「鎖方向 / 強制旋轉」——WCAG 1.3.4（Level AA）

- **WCAG 2.1 SC 1.3.4 Orientation（AA）**：內容**不得**限制只能用單一方向（portrait 或 landscape），除非該方向是「本質必要」（essential）。
- 為什麼是規範而非偏好：身障使用者常把裝置**固定**在輪椅 / 床邊支架上的單一方向；強制旋轉等於把他們擋在門外。
- 「我們的設計在直向比較好看」**不是**合法的鎖定理由。
- 合法例外（essential）：鋼琴 App、投影簡報、VR、銀行支票拍照——**SCCD 沒有任何頁面屬於此類**。
- 影響面：ADA、Section 508、EN 301 549、歐洲無障礙法案（EAA）都引用此條。

> 結論：`atlas` / `create` 等頁可以給「轉成橫向體驗更好」的**提示**，但**不能封鎖直向內容**。提示 ≠ 封鎖。

---

## 3. 橫向手機真正的問題是「高度」，不是寬度

橫向手機典型尺寸（CSS px，含瀏覽器 chrome 前的可視高更矮）：

| 機型 | 橫向尺寸（寬 × 高） | 落入本專案哪個斷點？ |
|---|---|---|
| iPhone SE | 667 × 375 | 寬 < 768 → **吃手機版** |
| iPhone 14 / 15 | 844 × 390 | 寬 ≥ 768 → **吃桌面版** ⚠️ |
| iPhone 15 Pro Max | 932 × 430 | 寬 ≥ 768 → **吃桌面版** ⚠️ |
| Pixel 7 / 多數 Android | ~915 × 412 | 寬 ≥ 768 → **吃桌面版** ⚠️ |

### ⚠️ 本專案的「斷點陷阱」（最重要的一個發現）

本專案是 **desktop-first，`md:` = 768px 起算桌面**（見 CLAUDE.md RWD 規範）。
**多數現代手機橫向時寬度 ≥ 768px → 直接套用「桌面版」版面，卻塞進一個只有 ~390–430px 高的視窗。**

後果：
- 桌面 `h-screen`（= `100vh`）hero 是為 800–1000px 高設計的，被壓進 390px。
- 桌面 H1 = 8rem（128px），在 390px 高的視窗裡佔掉三分之一。
- 滿版居中 hero（`flex items-center`）內容溢出上下緣被切。

而較舊 / 小機（如 SE，寬 667 < 768）則**吃手機版**，但手機版是為「窄而高」直向設計的，在「寬而矮」橫向同樣難看。

**兩種情況的共同解法：用「高度」當條件，不要用寬度。**
`@media (orientation: landscape) and (max-height: 500px)` 一次涵蓋上述兩種（不論它落在哪個寬度斷點），在這層把滿版高度拆掉、字級縮小、改自然流。

## 4. 最佳實踐（全部可重用，別自寫）

### 4.1 用瀏覽器原生動態視窗單位 `svh` / `dvh` / `lvh` 取代 `vh`

`100vh` 在手機會以「工具列收起後的最大高度」計算 → 矮橫向 + 瀏覽器 chrome 下會溢出。改用：

- **`svh`（small viewport height）**：以「工具列展開時的最小可視高」算 → **滿版 section 首選**，保證一進來就完整可見、不溢出。
- **`dvh`（dynamic）**：即時跟著工具列伸縮，但會在捲動時重排重繪 → 只在真的需要即時貼合時用。
- **`lvh`（large）**：最大可視高（= 舊 `vh` 行為）。
- 支援度：Chrome 108+ / Safari 15.4+ / Firefox 101+（>90% 使用者）。**保留 `vh` 當 fallback**（先寫 `vh` 再寫 `svh`，舊瀏覽器吃前者）。

```css
.hero {
  min-height: 100vh;   /* fallback */
  min-height: 100svh;  /* 現代瀏覽器：保證初始可見 */
}
```

> 本專案 `create.css:1165-1166` 已經在用 `100dvh` 當第二行 fallback——技術已在 codebase 內，只是沒全站推開。

### 4.2 「矮橫向」媒體查詢（業界通用片語）

```css
@media (orientation: landscape) and (max-height: 500px) {
  /* 只在矮橫向生效；不影響直向手機、也不影響桌面（桌面高度 > 500px）*/
}
```

`max-height: 500px` 同時命中「吃桌面版的寬橫向手機」與「吃手機版的小橫向手機」，是涵蓋面最廣又不誤傷桌面的閾值。

### 4.3 ✅ 本專案直接可用：Tailwind 內建（已是 3.4.18）

確認 `package.json` → **tailwindcss ^3.4.18**，以下原生 utility 直接可用，**不需要寫任何自訂 CSS**：

- **`landscape:` / `portrait:` variant**（v3.0+）：`landscape:hidden`、`landscape:flex-row`…
- **`h-svh` / `h-dvh` / `h-lvh` / `min-h-svh` / `min-h-dvh`**（v3.4+）：直接換掉 `h-screen` / `min-h-screen`。

這就是「用別人寫好的」最乾淨路徑——把 HTML 的 `h-screen` 換成 `h-svh`、要矮橫向特例就加 `landscape:` 前綴，**省掉一整層自寫 media query**。

```html
<!-- 之前 -->
<section class="h-screen flex items-center">
<!-- 之後：滿版用 svh + 矮橫向改成自然高、頂部對齊 -->
<section class="h-svh landscape:h-auto landscape:min-h-svh flex items-center landscape:items-start">
```

> 改 Tailwind class 後**要重新 build**（`npm run build:css`，見 package.json）。
> 純 `css/components/*` 的自訂規則改動，若是 router 動態載入的 page CSS（library/atlas/create/alumni）則不需 build；進 `input.css` bundle 的要 build。

### 4.4 例外頁面（方向「本質需要」）只給提示、不封鎖

- `atlas`（星雲 living textile）、`create`（p5 generator）這類重互動頁，橫向體驗確實較好。
- 做法：保留 / 沿用既有「橫向轉向提示」（commit `727ea96` 已為 atlas 做過），但**必須是可略過的提示**，直向時內容仍可操作（atlas 已改 list-only 直向版即符合此精神）。
- **不可**用 pleaserotate.js 那種「整頁蓋掉直到你轉向」的封鎖式做法（WCAG 1.3.4 違規）。

## 5. 本專案風險盤點（橫向手機會壞的地方，file:line）

> 共同根因：滿版 `h-screen` / `100vh` 為高螢幕設計，矮橫向會壓壞；且寬 ≥768 的橫向手機會誤吃桌面版。

| 區域 | 位置 | 矮橫向問題 |
|---|---|---|
| 首頁 hero / WATCH 區 | `index.html:35`（`h-screen` flex-end） | 滿版底部對齊內容在 390px 高被切 |
| about hero / 多個 `min-height:100vh` 區段 | `pages/about.html:20,217,368,388,398` | 連續滿版段落，矮橫向每段都過高 |
| 內頁 hero（共用範式） | `admission/activities/faculty/curriculum.html` 的 `min-h-screen md:h-screen` | 寬≥768 吃 `md:h-screen` → 桌面 hero 塞進矮窗 |
| alumni hero | `pages/alumni.html:25`（`h-screen` 居中） | 居中內容上下溢出 |
| degree-show-detail hero | `pages/degree-show-detail.html:30,67` | 同上 + 字卡靠圖底的 min-height 計算失準 |
| **atlas（鎖頁）** | `pages/atlas.html:12`（`body.overflow-hidden h-screen`） | 整頁鎖死滿版，矮橫向最糟；已有 list-only 直向版可沿用 |
| **library（鎖頁）** | `pages/library.html:18,26`；`library.css:760` | `body` 鎖 `h-screen`、panel `calc(100vh - header)` |
| **create（鎖頁）** | `pages/create.html:21`；`create.css:38,140,1165` | `100vh - header` 工作區；已有 `100dvh` fallback |
| hero 旋轉 bbox 計算 | `hero.css:98-100,145`（`100vh` 換算旋轉範圍） | 矮橫向下 `100vh` 變小，旋轉標題 bbox 算錯被切 |
| footer | `footer.css:23`（`height:100vh`） | 滿版 footer 在矮橫向過高 |
| legal sticky 左欄 | `legal.css:15,18,24`（`100vh` sticky） | sticky 欄 = 整個矮螢幕，內文無空間 |
| 404 散佈 stage | `pages/404.html:31`（`100vh` 絕對定位散佈） | 卡片以直向比例散佈，矮橫向重疊 |
| 共用 hero CSS | `variables.css:331`（hero 高度註解）、`color.css:73`（`section.h-screen`） | 全站 hero 高度來源，集中改點 |

**好消息**：以上 80% 共用同一個 `h-screen` / `100vh` hero 模式 → **一層全域「矮橫向」CSS 就能一次蓋掉大部分**，不用逐頁重寫。

## 6. 建議落地步驟（reuse-first，分階段）

### Phase 0 — 全域「矮橫向」防護層（CP 值最高，先做這個）
新增 `css/layout/landscape.css`，在 `input.css` 末段 `@import`（cascade 後者勝），內容只做三件事：

```css
@media (orientation: landscape) and (max-height: 500px) {
  /* 1. 滿版 hero 不再強制滿高，改自然流、頂部對齊（避免內容被上下切） */
  main > section.h-screen,
  main > section[class*="min-h-screen"] {
    min-height: 100svh;
    height: auto;
    align-items: flex-start;
  }
  /* 2. 收掉過大的 hero 標題（桌面 8rem 在矮窗太高），用既有變數降一級 */
  /* 3. footer 等滿版區改 svh */
}
```

> 數值與選擇器以實測為準（見 Phase 3）。這一層理論上修好 index / about / 內頁 hero 群 / alumni / footer 一大批。

### Phase 1 — 把 `h-screen` → `h-svh`（Tailwind 原生，需 build）
逐頁把 HTML 的 `h-screen` / `min-h-screen` 換成 `h-svh` / `min-h-svh`，矮橫向特例用 `landscape:` 前綴。改完 `npm run build:css`。

### Phase 2 — 三個鎖頁（atlas / library / create）個別處理
這三頁 `body` 鎖 `overflow-hidden h-screen`，工具列 / panel 用 `100vh - header` 算高 → 換成 `svh`，並確認矮橫向控制列不擋內容。create 已有 `dvh` 基礎。

### Phase 3 — 例外頁提示（atlas / create）對齊 WCAG
確認 atlas 既有「橫向轉向提示」是**可略過提示**而非封鎖；直向 list-only 版要能完整操作。

### Phase 4 — 實測驗證
用 memory 既有的 playwright recipe（`reference_playwright_animation_probe_recipe.md`）在橫向尺寸截圖比對：
- **844 × 390**（吃桌面版的寬橫向）← 最關鍵
- **932 × 430**（大機橫向）
- **667 × 375**（吃手機版的小橫向）

## 7. 驗證 checklist

- [ ] 矮橫向下無內容被上下緣切掉、可正常捲動
- [ ] 寬 ≥768 橫向手機（吃桌面版）hero 不爆高、字級合理
- [ ] **沒有任何頁面鎖定方向 / 強制旋轉**（WCAG 1.3.4）
- [ ] 直向版完全不受影響（desktop-first 鐵則：手機改不影響桌面、橫向改不影響直向）
- [ ] 真正桌面（高度 > 500px）不被 `max-height:500px` 規則波及
- [ ] 改 Tailwind class 的頁面已 `npm run build:css`
- [ ] notch 機型橫向側邊用 `env(safe-area-inset-*)`（如有絕對定位貼邊元素）
